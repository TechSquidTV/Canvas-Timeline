## Breaking change warning

The media adapters now accept one app-resolved media choice for each logical
timeline source. The old flat descriptors, duplicate `sourceId` fallback
convention, adapter-owned original/proxy selection, `durationBySourceId`, and
blocking `resumeClock()` contract have been removed. There are no compatibility
aliases.

Media contracts such as `TimelineMediaSyncAdapter`, `TimelineMediaSourceTiming`,
and `TimelineMediaError` now come from the framework-free core package. HTML
React hooks moved to the adapter's `./react` export, matching the Mediabunny
package boundary.

```tsx
import { useHTMLTimelineMedia } from '@techsquidtv/canvas-timeline-html-media-adapter/react';
```

The common case stays concise: `{ sourceId, input }`. Use `fallbacks` only for
equivalent transport or cache alternatives. Keep original, proxy, preview, and
export policy in the application's media library.

## Migrate HTML media sources

Replace the source record with an array of logical sources. Put the normal URL,
blob, or file in `input` and optional load failovers in `fallbacks`.

```tsx
const sources = [
  {
    sourceId: 'intro',
    input: cdnUrl,
    fallbacks: [fallbackFile],
  },
];

const media = useHTMLTimelineMedia({
  ref: mediaRef,
  sources,
  layers,
});
```

Read `sourceStateById` to inspect lifecycle status, `selectedInputIndex`, and
ordered attempts. Use `media.adapter.retrySource(sourceId)` to retry inputs or
`media.adapter.replaceSource(source)` to replace the resolved media without
recreating the adapter. Both commands return an explicit `configured` or
`ready` success state instead of a boolean.

React hooks reconcile sources by value, so an inline array of ordinary URL
descriptors no longer disposes and recreates the adapter on every render. Keep
input factories, track selectors, custom URL option objects, and other callback
configuration stable because their identity represents executable policy.

## Migrate Mediabunny sources

Pass a normal URL string, `URL`, `Request`, `Blob`, or `File` directly as
`input`. Use an object descriptor when you need formats, `UrlSource` options, a
supplied input, or an input factory. Put equivalent direct-media failover in
`fallbacks`.

```tsx
const sources = [
  {
    sourceId: 'camera-a',
    input: {
      kind: 'url',
      url: manifestUrl,
      formats: HLS_FORMATS,
      urlSourceOptions: { maxCacheSize: 32 * 1024 * 1024 },
    },
    fallbacks: [
      {
        kind: 'url',
        url: mp4Url,
      },
    ],
  },
];
```

Use `selectTracks` when primary tracks are not the desired video or audio
tracks. Source state now includes detected timing, dimensions, frame rate, and
audio metadata alongside input-attempt diagnostics.

Mediabunny sources are registered immediately and loaded only when their clips
become active. `ready` therefore means the adapter can resolve configured media;
it does not mean every project asset has been opened. Use
`preloadSource(sourceId)` for likely next clips and `unloadSource(sourceId)` to
release decoder resources while keeping the definition registered.

## Model editing proxies in the media library

A proxy is an application-level media relationship, not a decoder fallback. An
editor may select it for scrubbing while retaining the original for high-quality
preview, conform, relink, and export. Model those purposes in media-library state
and resolve one adapter source from the current editing policy.

```tsx
const cameraA = {
  sourceId: 'camera-a',
  original: { input: originalFile },
  proxies: {
    editing: {
      input: proxyUrl,
      fallbacks: [cachedProxy],
      timing: {
        sourceTimeSeconds: cameraTimestamp,
        mediaTimeSeconds: 0,
      },
    },
  },
} as const;

function resolvePreviewSource(useProxy: boolean) {
  const resolved = useProxy ? cameraA.proxies.editing : cameraA.original;
  return { sourceId: cameraA.sourceId, ...resolved };
}

await media.adapter.replaceSource(resolvePreviewSource(true));
```

The adapter does not need to know why that media was chosen. The `timing` anchor
maps logical source time to the resolved media time and is needed only when a
proxy or transcode uses a different container timestamp origin. Switching back
uses `replaceSource(resolvePreviewSource(false))`; timeline clips keep the same
`sourceId` throughout.

This keeps policy extensible without changing adapter types: an application can
also resolve offline cache entries, HLS/direct delivery, color-managed review
media, remote collaboration previews, or export originals through the same
source boundary.

## Update transport and audio controls

Pass the same `playbackOptions` used by engine playback when an external media
clock drives the timeline. External ticks now enforce in/out boundaries and
looping through the engine's shared playback policy.

Clock activation is intentionally non-blocking. Call `requestClockActivation()`
when transport starts and observe audio activation state separately from video
transport. Use `setVolume()` and `setMuted()` for runtime gain changes; neither
operation reloads sources.

Caller-supplied `AudioContext` instances remain caller-owned. The Mediabunny
adapter closes only contexts it creates, and it does not create an audio graph
for sources without a decodable audio track.

Error handlers now receive `TimelineMediaError` with a stable `reason` and a
human-readable `message`:

```tsx
useMediabunnyTimelineMedia({
  sources,
  layers,
  onError: (error) => reportPreviewFailure(error.reason, error.message),
});
```

At the low-level `useTimelineMediaPlayback` boundary, provide `loop` as the
clock-realignment callback. Its presence enables looping, so an external clock
cannot be configured to loop without a way to seek back to the range start.
